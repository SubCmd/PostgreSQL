## 1. 그룹 함수 — 개념 정리
 
### 1-1. 용어 풀이

| 용어 | 풀어서 설명 | PM/분석가 비유 |
|---|---|---|
| **ROLLUP(A, B)** | 상세 집계에 더해, 나열 **순서대로** 계층을 하나씩 걷어 올리며(roll up) 소계·총계 행을 **추가 생성**함 | 엑셀 피벗 테이블에서 "소계 표시 + 총합계 표시"를 켠 상태 |
| **CUBE(A, B)** | 나열된 컬럼의 **모든 조합 방향**으로 소계를 생성함 (A 소계, B 소계 모두) | 크로스탭 보고서에서 행 방향·열 방향 margin을 전부 붙인 상태 |
| **GROUPING SETS(...)** | 원하는 집계 단위만 **골라 담는** 방식. ROLLUP/CUBE는 사실 이것의 축약형임 | 뷔페식 — 필요한 소계만 접시에 담음 |
| **GROUPING(col)** | 해당 행의 NULL이 "소계/총계 때문에 생긴 NULL"이면 **1**, 실제 데이터 NULL이면 **0**을 반환함 | GA4 리포트의 "(not set)" 과 진짜 결측을 구분하는 플래그 |
| **초집계 행 (super-aggregate row)** | ROLLUP/CUBE가 추가로 만들어낸 소계·총계 행. 걷어 올린 컬럼 자리는 NULL로 표시됨 | — |

ROLLUP(a, b)
(a, b) : 전체 세부 그룹
(a) : 기준 소계
() : 전체 합계

CUBE(a, b)
(a, b) : 전체 세부 그룹
(a) : a 기준 소계
(b) : b 기준 소계
() : 전체 합계

GROUPING SETS((a, b), (a), (b), ())
: 집계 단위(Set)를 사용자가 직접 명시
ROLLUP(a, b) = GROUPING SETS ((a, b), (a), ())
CUBE(a, b) = GROUPING SETS ( (a, b), (a), (b), ())

### 1-2. 핵심: 행 수 계산 공식 (시험 최다 출제)
 
공통 표기: `|A,B|` = (A,B) 조합의 종류 수, `|A|` = A 값의 종류 수
 
| 구문 | 행 수 공식 | 동일한 GROUPING SETS 표현 |
|---|---|---|
| `ROLLUP(A, B)` | `|A,B| + |A| + 1` | `GROUPING SETS((A,B), (A), ())` |
| `ROLLUP(A, B, C)` | `|A,B,C| + |A,B| + |A| + 1` | `GROUPING SETS((A,B,C), (A,B), (A), ())` |
| `CUBE(A, B)` | `|A,B| + |A| + |B| + 1` | `GROUPING SETS((A,B), (A), (B), ())` |
| `GROUPING SETS(A, B)` | `|A| + |B|` — **총계 없음 주의** | 그 자체 |
| `ROLLUP((A, B))` | `|A,B| + 1` — 괄호로 묶으면 **한 덩어리** 취급 | `GROUPING SETS((A,B), ())` |
| `A, ROLLUP(B)` | `|A,B| + |A|` — GROUP BY에 고정된 A는 항상 유지됨 | `GROUPING SETS((A,B), (A))` |
 
> `()` 는 "아무 것도 기준으로 잡지 않음" = **총계 1행**을 의미함.

### 1-3. ✅ Good vs ❌ Bad

**① 소계 NULL vs 데이터 NULL 구분**
 
```sql
-- ❌ Bad: NVL로 소계 라벨을 만들면, 실제 데이터가 NULL인 행도 '전체'로 둔갑함
SELECT NVL(CHANNEL, '전체') AS CHANNEL, SUM(AMT)
FROM ORDERS
GROUP BY ROLLUP(CHANNEL);
 
-- ✅ Good: GROUPING()으로 "초집계 행"만 정확히 라벨링함
SELECT CASE WHEN GROUPING(CHANNEL) = 1 THEN '전체'
            ELSE CHANNEL END AS CHANNEL,
       SUM(AMT)
FROM ORDERS
GROUP BY ROLLUP(CHANNEL);
-- PG: 동일 문법 그대로 동작함
```

**② 인수 순서**
 
```sql
-- ❌ Bad 인식: "ROLLUP은 순서 바꿔도 행 수가 같다"
--    ROLLUP(A,B) = |A,B|+|A|+1  vs  ROLLUP(B,A) = |A,B|+|B|+1
--    → |A| ≠ |B| 이면 행 수가 달라짐 (계층 방향이 다름)
 
-- ✅ Good 인식: "CUBE는 순서 무관 (모든 방향 소계라서), ROLLUP은 순서 의존"
```
 
**③ ROLLUP을 UNION ALL로 흉내내기 (시험 단골 등가 표현)**
 
```sql
-- ROLLUP(CHANNEL) 은 아래와 결과가 동일함 (정렬 제외)
SELECT CHANNEL, SUM(AMT) FROM ORDERS GROUP BY CHANNEL
UNION ALL
SELECT NULL,    SUM(AMT) FROM ORDERS;
-- 시험에서 "다음 중 위 SQL과 결과가 같은 것은?" 유형으로 출제됨
```

## 2. 윈도우 함수 — 개념 정리
 
### 2-1. 핵심 구조

```sql
함수(인자) OVER ( PARTITION BY 그룹기준 -- 생략 시 전체가 합 그룹
                 ORDER BY 정렬기준     -- 순위·누적 계산의 방향
                 ROWS|RANGE BETWEEN ... ) -- 계산 범위 (윈도우 프레임)
```

> **GROUP BY와 차이**: GROUP BY는 행을 **접어서** 그룹당 1행으로 만들고, 윈도우 함수는 행을
**그대로 둔 채** 옆에 집계 컬럼을 붙임.
> (Amplitude에서 유저별 이벤트 로그를 유지하면서 누적 세션 수 컬럼을 추가하는 것과 같은)


### 2-2. 3대 함정

| 함정 | 내용 |
|---|---|
| **① RANK 건너뜀** | 100, 90, 90, 80 → `RANK`: 1,2,2,**4** (공동 순위만큼 건너뜀) / `DENSE_RANK`: 1,2,2,**3** / `ROW_NUMBER`: 1,2,3,4 (동점도 강제로 다른 번호) |
| **② LAST_VALUE 기본 윈도우** | ORDER BY가 있으면 기본 프레임이 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` → LAST_VALUE가 "현재 행까지의 마지막" = 자기 자신을 반환함. 진짜 마지막 값을 원하면 `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` 명시 필요 |
| **③ ROWS vs RANGE** | `ROWS`는 물리적 행 단위, `RANGE`는 **ORDER BY 값이 같은 행을 한 묶음**으로 처리 → 동일 값 구간에서 누적값이 달라짐 (아래 Lv.2-B3) |

### 2-3. ✅ Good vs ❌ Bad — Oracle Top N
 
```sql
-- ❌ Bad: WHERE의 ROWNUM은 "정렬 전"에 매겨짐 → 아무 3행이나 뽑아서 정렬함
SELECT * FROM EMP
WHERE ROWNUM <= 3
ORDER BY SAL DESC;
 
-- ✅ Good 1: 인라인 뷰로 먼저 정렬한 뒤 ROWNUM 적용
SELECT * FROM (SELECT * FROM EMP ORDER BY SAL DESC)
WHERE ROWNUM <= 3;
 
-- ✅ Good 2: 표준 문법 (Oracle 12c+, PG 동일)
SELECT * FROM EMP ORDER BY SAL DESC
FETCH FIRST 3 ROWS ONLY;
-- PG: LIMIT 3 도 가능하나 SQLD 정답 선지는 위 두 형태로 출제됨
```
