## 1. 서브쿼리란?
**서브쿼리(subquery)**는 다른 쿼리 안에 들어간 `SELECT`문으로, 들어가는 위치에 따라 역할이 다름

| 위치 | 이름 | 반환 형태 | 용도 | 
| --- | ---- | -------- | ---- | 
| `SELECT`절 | 스칼라 서브쿼리 | 단일 값(1행 1열) | 각 행에 계산값 붙이기 |
| `FROM`절 | 인라인 뷰 | 테이블(여러 행·열) | 중간 결과를 테이블처럼 | 
| `WHERE`절 | 중첩/상관 서브쿼리 | 단일 값 또는 목록 | 조건 필터링 |

---

## 2. 스칼라 서브쿼리 (단일 값)

```sql
-- SELECT 절: 각 상품 가격을 '전체 평균가'와 함께 표시
SELECT
    name,
    price,
    (SELECT ROUND(AVG(price)) FROM products) AS 전체평균가,
    price - (SELECT AVG(price) FROM products) AS 평균과의차이
FROM products;

-- WHERE 절: 평균보다 비싼 상품만
SELECT name, price FROM products
WHERE price > (SELECT AVG(price) FROM products)
ORDER BY price DESC;
```

> ⚠️ 스칼라 서브쿼리는 **반드시 1행 1열**을 반환해야 한다. 여러 행이 나오면 에러.

---

## 3. 인라인 뷰 (FROM 절 서브쿼리)

```sql
-- 회원별 주문건수를 먼저 구하고(인라인 뷰), 그중 2건 이상만 평균내기
SELECT ROUND(AVG(주문건수), 1) AS 평균주문건수
FROM (
    SELECT user_id, COUNT(*) AS 주문건수
    FROM orders
    WHERE status = 'completed'
    GROUP BY user_id
) AS user_orders
WHERE 주문건수 >= 2;
```

> 💡 "집계 결과를 또 집계"하거나 "집계 결과를 필터"할 때 유용. 다만 인라인 뷰가 중첩되면 읽기 어려워진다
> → 이걸 깔끔하게 푸는 게 7장 CTE.

## 4. WHERE 절 서브쿼리 - IN / 비교

```sql
-- 완료 주문을 한 적 있는 회원의 정보 (IN: 목록 매칭)
SELECT name, email FROM users
WHERE user_id IN (
    SELECT user_id FROM orders WHERE status = 'completed'
);

-- 가장 비싼 상품과 같은 카테고리의 상품들
SELECT name, category, price FROM products
WHERE category = (
    SELECT category FROM products ORDER BY price DESC LIMIT 1
);
```

## 5. 상관 서브쿼리 & EXISTS

### 5-1. 상관 서브쿼리 (Correlated)
바깥 쿼리의 각 행을 참조하며 행마다 실행되는 서브쿼리.

```sql
-- 각 회원이, 자기 채널의 평균 가입일보다 늦게 가입했는지?
SELECT u.name, u.channel, u.signup_date
FROM users u
WHERE EXTRACT(EPOCH FROM u.signup_date) > (
    SELECT AVG(EXTRACT(EPOCH FROM u2.signup_date))
    FROM users u2
    WHERE u2.channel = u.channel        -- 바깥 u를 참조 → 상관
);
```

### 5-2. EXISTS / NOT EXISTS (존재 여부)

```sql
-- EXISTS: 완료 주문이 '존재하는' 회원
SELECT u.name FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.user_id = u.user_id AND o.status = 'completed'
);

-- NOT EXISTS: 한 번도 주문하지 않은 회원 (LEFT JOIN + IS NULL 대안)
SELECT u.name FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.user_id
);
```

> 💡 `EXISTS`는 "있는지 없는지"만 보므로 `SELECT 1`로 충분(값은 안 봄).
> 매칭되는 첫 행을 찾으면 즉시 멈춰서 `IN`보다 효율적인 경우가 많음.
> **"존재/부재" 판단은 `EXISTS`/`NOT EXISTS`** 가 의미도 명확하고 NULL 함정도 적다.
 
> ⚠️ **`NOT IN` + NULL 함정**: 서브쿼리 결과에 NULL이 하나라도 있으면 `NOT IN`은 **아무 행도 반환하지 않는다**.
> 부재 판단엔 `NOT EXISTS`가 안전하다.

> 즉, `EXISTS`는 **데이터 존재 여부**에 따라 달라짐.
> `NOT IN`은 `NULL`이 포함될 가능성이 단 1%라도 있다면 절대로 안됨.

## 6. ANY / ALL

```sql
-- ANY: 서브쿼리 결과 중 '하나라도' 만족 (electronics 가격들 중 최소보다 크면)
SELECT name, price FROM products
WHERE price > ANY (
    SELECT price FROM products WHERE category = 'electronics'
); -- = electronics 최저가보다 비싼 상품

-- ALL: 서브쿼리 결과 '모두'보다 (모든 electronics보다 비싸면)
SELECT name, price FROM products
WHERE price > ALL (
    SELECT price FROM products WHERE category = 'electronics'
); -- = electronics 최고가보다 비싼 상품
```

> `> ANY (목록)` = 목록 최솟값보다 큼,
> `> ALL (목록)` = 목록 최댓값보다 큼.
> `= ANY`는 `IN`과 같다.

## 7. CTE(Common Table Expression) - (WITH) 쿼리를 단계로 쪼개기 ⭐
`WITH 이름 AS (...)`로 이름 붙인 임시 결과를 만들어 본 쿼리에서 테이블처럼 쓴다.
인라인 뷰보다 훨씬 읽기 좋음.

- `WITH` 구문의 주요 기능과 장점
1. 가독성 향상 : 쿼리를 위에서 아래로 순차적으로 읽을 수 있게 구조화
2. 코드 재상용성 : 동일한 서브쿼리를 반복해서 적을 필요가 없음
3. 유지보수 용이성 : 복잡한 비즈니스 로직을 단계별로 분리하여 정의
4. 재귀쿼리 지원 : 계층형 데이터를 손쉽게 조회 가능

```sql
-- 회원별 완료주문 집계를 CTE로 만들고, 본 쿼리에서 활용
WITH user_orders AS (
    SELECT user_id, COUNT(*) AS 주문건수
    FROM orders
    WHERE status = 'completed'
    GROUP BY user_id
)
SELECT u.name, COALESCE(uo.주문건수, 0) AS 주문건수
FROM users u
LEFT JOIN user_orders uo ON u.user_id = uo.user_id
ORDER BY 주문건수 DESC;
```

### 7-1. 다중 CTE 체이닝 (Phase 4 매출 쿼리 리팩터링)
Phase4의 길었던 "채널별 매출 리포트"를 CTE로 단계화하면 의도가 또렷해짐.

```sql
WITH completed_orders AS (          -- 1단계: 완료 주문만
    SELECT order_id, user_id
    FROM orders
    WHERE status = 'completed'
),
order_revenue AS (                  -- 2단계: 주문별 매출
    SELECT co.order_id, co.user_id,
           SUM(oi.quantity * oi.unit_price) AS 매출
    FROM completed_orders co
    JOIN order_items oi ON co.order_id = oi.order_id
    GROUP BY co.order_id, co.user_id
)
SELECT                              -- 3단계: 채널별 집계
    u.channel,
    COUNT(DISTINCT orv.order_id)    AS 주문수,
    SUM(orv.매출)                   AS 총매출,
    ROUND(AVG(orv.매출))            AS 객단가
FROM order_revenue orv
JOIN users u ON orv.user_id = u.user_id
GROUP BY u.channel
ORDER BY 총매출 DESC;
```

> 💡 **CTE의 진짜 가치**
> ① 가독성(단계마다 이름), ② 같은 중간결과 재사용, ③ 디버깅 쉬움(CTE 하나씩 떼어 실행). 복잡한 분석 쿼리는 CTE로 짜는 게 실무 표준임.
> fan-out도 단계를 분리하면 통제하기 쉽다(주문별로 먼저 매출을 묶고 채널 집계).

## 8. 재귀 CTE (Recursive)
`WITH RECURSIVE`는 자기 자신을 참조하며 반복함.
조직도(상사-부하), 카테고리 트리, 연속 날짜/숫자 생성에 씀

```sql
-- 날짜 스파인 생성: 2024-01-01 ~ 2024-03-31 모든 날짜
WITH RECURSIVE date_spine AS (
    SELECT '2024-01-01'::DATE AS dt     -- 시작점 (anchor)
    UNION ALL
    SELECT dt + 1                       -- 재귀: 하루씩 더함
    FROM date_spine
    WHERE dt < '2024-03-31'             -- 종료 조건 (필수!)
)
SELECT dt FROM date_spine;
```

> 💡 **PM 관점**
> 날짜 스파인은 시계열 분석의 필수 도구.
> "주문이 0건인 날"도 차트에 0으로 표시하려면 연속 날짜 테이블을 만들어 `LEFT JOIN` 해야 한다.
> (PostgreSQL은 `generate_series('2024-01-01','2024-03-31', '1 day')`로 더 간단히도 가능 — 재귀 개념은 트리 구조에서 진가 발휘)
 
> ⚠️ 재귀 CTE는 **종료 조건이 없으면 무한 루프**. 반드시 멈추는 조건을 넣는다.

## 9. 서브쿼리 vs. JOIN vs. CTE - 언제 무엇을?

| 상황 | 권장 |
| ---- | ---- | 
| 단일 값 비교 (평균보다 큼) | 스칼라 서브쿼리 |
| 존재/부재 판단 | `EXISTS`/`NOT EXISTS`|
| 여러 테이블 컬럼을 함께 보기 | JOIN | 
| 다단계 변환·집계, 가독성 중요 | CTE(`WITH`) | 
| 같은 중간결과 여러 번 사용 | CTE | 

> 💡 정답은 하나가 아니다.
> 같은 결과를 서브쿼리·JOIN·CTE로 다 짤 수 있는 경우가 많다.
> **"읽는 사람이 의도를 빨리 이해하는가"** 가 실무 기준. 복잡해지면 CTE로 가라.

## 10. 종합 예시 (실무형)

```sql
-- "객단가 상위 회원 분석: 평균 객단가보다 높은 회원 추출"
WITH user_revenue AS (
    SELECT
        o.user_id,
        SUM(oi.quantity * oi.unit_price) AS 총매출,
        COUNT(DISTINCT o.order_id)       AS 주문수
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    WHERE o.status = 'completed'
    GROUP BY o.user_id
),
user_aov AS (
    SELECT user_id, 총매출, 주문수,
           ROUND(총매출::NUMERIC / NULLIF(주문수, 0)) AS 객단가
    FROM user_revenue
)
SELECT u.name, ua.주문수, ua.총매출, ua.객단가
FROM user_aov ua
JOIN users u ON ua.user_id = u.user_id
WHERE ua.객단가 > (SELECT AVG(객단가) FROM user_aov)    -- 평균 객단가 초과
ORDER BY ua.객단가 DESC;
```
