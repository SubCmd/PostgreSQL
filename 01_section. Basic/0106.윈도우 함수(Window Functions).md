# Phase 1-6: 윈도우 함수 (Window Functions) ⭐

> **목표**: 행을 합치지 않고 **유지한 채**, 그룹 내 순위·누적·이동평균·직전값을 계산한다. "회원별 첫 주문", "일별 누적 매출", "전월 대비 성장률", "세션 기반 퍼널" — Growth 분석의 가장 강력한 도구.

## 1. GROUP BY vs 윈도우 함수 (핵심 직관)

```
GROUP BY: 그룹을 '한 줄로 압축'         윈도우 함수: 행을 '유지'하며 계산 추가
┌──────┬─────┐                          ┌──────┬─────┬──────────┐
│channel│합계 │                         │ name │price│ 채널평균  │
├──────┼─────┤                          ├──────┼─────┼──────────┤
│paid  │ 300 │   ← 3행이 1행으로         │김민준 │ 100 │   100    │  ← 행 그대로
│organic│200 │                          │이서연 │ 100 │   100    │
└──────┴─────┘                          │박지후 │ 100 │   100    │
                                        └───────┴─────┴──────────┘
```

> 💡 **핵심**
> `GROUP BY`는 "그룹당 한 행"으로 줄임.
> 윈도우 함수는 **원본 행을 그대로 두고** 옆에 집계/순위 컬럼을 덧붙인다.
> "각 상품과 그 카테고리 평균을 한 줄에 같이 보고 싶다"면 윈도우 함수.

## 2. 기본구조: OVER / PARTITION BY / ORDER BY
```sql
함수() OVER (
    PARTITION BY 그룹기준       -- 어떻게 묶을지 (생략 시 전체가 한 그룹)
    ORDER BY 정렬기준           -- 그룹 내 순서 (순위·누적·LAG에 필수)
)
```

```sql
-- 각 상품의 가격 + 같은 카테고리의 평균가를 한 줄에
SELECT
    name,
    category,
    price,
    ROUND(AVG(price) OVER (PARTITION BY category)) AS 카테고리평균,
    price - AVG(price) OVER (PARTITION BY category) AS 평균과의차이
FROM products
ORDER BY category, price DESC;
```

> `PARTITION BY category`= 카테고리별로 창(window)을 나눠 각 창 안에서 평균을 계산.
> `GROUP BY`와 달리 행이 사라지지 않는다.

`GROUP BY` : 지정한 칼럼을 기준으로 데이터를 완전히 뭉쳐버리기 때문에 행의 개수가 줄어들고 개별 행의 상세 정보가 사라짐.
`PARTITION BY` : 논리적인 구역(창)만 나누어 계산할 뿐, 행을 합치지 않으므로 개별 행의 상세 정보를 유지하면서 집계 값을 추가할 수 있음.

## 3. 순위 함수 - ROW_NUMBER / RANK / DENSE_RANK / NTILE

```sql
-- 카테고리별 가격 순위 (세 함수 차이 비교)
SELECT
    name, category, price,
    ROW_NUMBER() OVER (PARTITION BY category ORDER BY price DESC) AS 행번호,
    RANK()       OVER (PARTITION BY category ORDER BY price DESC) AS 순위,
    DENSE_RANK() OVER (PARTITION BY category ORDER BY price DESC) AS 조밀순위
FROM products;
```

| 함수 | 동점 처리 | 예: 100,100, 90 |
| ---- | -------- | --------------- |
| `ROW_NUMBER` | 무조건 고유 번호 | 1, 2, 3 |
| `RANK` | 동점은 같은 순위, 다음은 건너뜀 | 1, 1, 3 |
| `DENSE_RANK` | 동점 같은 순위, 다음은 안 건너뜀 | 1, 1, 2 |

```sql
-- NTILE: N개 그룹으로 균등 분할 (RFM 점n수, 사분위에 활용)
SELECT
    name, price,
    NTILE(4) OVER (ORDER BY price DESC) AS 가격_사분위  -- 1=상위25%
FROM products;
```

### "각 그룹의 첫 번째" 뽑기 - ROW_NUMBER의 대표 패턴 ⭐

```sql
-- 회원별 '첫 주문'만 추출 (PARTITION BY user_id, ORDER BY 날짜, rn=1)
WITH ranked AS (
    SELECT
        user_id, order_id, order_date,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date) AS rn
    FROM orders
    WHERE status = 'completed'
)
SELECT user_id, order_id, order_date AS 첫주문일
FROM ranked
WHERE rn = 1;
```

> 💡 **최빈 실무 패턴**
> "그룹별 최신/최초 1건"은 `ROW_NUMBER() ... = 1`. 최근 로그인, 첫 구매, 최신 상태 등에 끝없이 쓰인다.

## 4. LAG / LEAD - 직전·다음 행 참조 (성장률의 핵심)

```sql
-- 월별 매출과 '전월 매출', 그리고 성장류
WITH monthly AS (
    SELECT
        DATE_TRUNC('month', o.order_date)::DATE AS 월,
        SUM(oi.quantity * oi.unit_price) AS 매출
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    WHERE o.status = 'completed'
    GROUP BY DATE_TRUNC('month', o.order_date)
)
SELECT
    월,
    매출,
    LAG(매출) OVER (ORDER BY 월) AS 전월매출,
    ROUND(
            (매출 - LAG(매출) OVER (ORDER BY 월))::NUMERIC
            / NULLIF(LAG(매출) OVER (ORDER BY 월), 0) * 100, 1
        ) AS 전월대비_성장률
FROM monthly
ORDER BY 월;
```

> `LAG(컬럼, n, 기본값)` = n행 앞의 값(기본 1).
> `LEAD`는 n행 뒤. 전월/전일 대비, 직전 주문과의 간격 계산에 필수.

## 5. 누적합 & 이동평균 (집계 윈도우)

```sql
-- 일별 매출 + 누적 매출(running total) + 3일 이동평균
WITH daily AS (
    SELECT
        o.order_date AS 날짜,
        SUM(oi.quantity * oi.unit_price) AS 일매출
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    WHERE o.status = 'completed'
    GROUP BY o.order_date
)
SELECT
    날짜,
    일매출,
    SUM(일매출) OVER (ORDER BY 날짜) AS 누적매출,           -- running total
    ROUND(AVG(일매출) OVER (
        ORDER BY 날짜 ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    )) AS 이동평균_3일                                      -- 최근 3일 평균
FROM daily
ORDER BY 날짜;
```

> 💡 `SUM() OVER (ORDER BY ...)`는 정렬 순서대로 누적.
> 이동평균은 프레임(`ROWS BETWEEN`)으로 "최근 N행"을 지정. 시계열 추세를 부드럽게 보는 표준 기법.

## 6. 프레임 (ROWS / RANGE BETWEEN)
`ORDER BY`가 있는 윈도우는 기본 프레임이 "시작~현재행"이다. 명시적으로 범위를 제어할 수 있음

```sql
함수() OVER (ORDER BY x ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)   -- 현재+앞 2행
함수() OVER (ORDER BY x ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)   -- 처음~현재(누적)
함수() OVER (ORDER BY x ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING)   -- 현재~끝
```

| 키워드 | 의미 |
|--------|------|
| `UNBOUNDED PRECEDING` | 파티션 시작 |
| `n PRECEDING` | n행 앞 |
| `CURRENT ROW` | 현재 행 |
| `n FOLLOWING` | n행 뒤 |
| `UNBOUNDED FOLLOWING` | 파티션 끝 |

> ⚠️ `ROWS`(물리적 행 수)와 `RANGE`(값 기준 범위)는 동점 처리가 다르다.
> 이동평균엔 보통 `ROWS`를 쓴다.

### FIRST_VALUE / LAST_VALUE

```sql
-- 각 카테고리에서 가장 비싼 상품명을 모든 행에 표시
SELECT
    name, category, price,
    FIRST_VALUE(name) OVER (
        PARTITION BY category ORDER BY price DESC
    ) AS 카테고리_최고가상품
FROM products;
```

> ⚠️ `LAST_VALUE`는 기본 프레임 때문에 의도와 다르게 동작하기 쉽다.
> "마지막 값"을 원하면 `ORDER BY 역순 + FIRST_VALUE`가 안전하거나, 프레임을 `UNBOUNDED FOLLOWING`까지 명시해야 한다.

## 7. 세션 기반 순차 퍼널 (LAG/LEAD 실전) ⭐
: Phase 3에서 미뤄둔 "순서를 고려한 퍼널"을 드디어 해결한다.

```sql
-- 같은 세션 안에서 각 이벤트의 '다음 이벤트'를 본다
SELECT
    session_id,
    user_id,
    event_name AS 현재이벤트,
    occurred_at,
    LEAD(event_name) OVER (
        PARTITION BY session_id ORDER BY occurred_at
    ) AS 다음이벤트
FROM events
ORDER BY session_id, occurred_at;

-- 응용: 'add_to_cart 다음에 곧바로 checkout으로 간' 전환 카운트
WITH seq AS (
    SELECT
        session_id, event_name,
        LEAD(event_name) OVER (PARTITION BY session_id ORDER BY occurred_at) AS 다음
    FROM events
)
SELECT
    COUNT(*) FILTER (WHERE event_name='add_to_cart' AND 다음='checkout') AS 장바구니_결제전환,
    COUNT(*) FILTER (WHERE event_name='add_to_cart') AS 장바구니_총
FROM seq;
```

> 💡 누적 퍼널(단순 카운트)과 달리, `LAG/LEAD`는 **세션 내 실제 이동 경로**를 본다.
> `FILTER (WHERE ...)`는 조건부 집계의 PostgreSQL 간결 문법(= `SUM(CASE WHEN ...)`).
> 진짜 전환 분석은 이 순서 정보가 핵심.

## 8. 종합 예시 (실무형) — 회원 구매 여정

```sql
-- "회원별 주문 시퀀스: 몇 번째 주문인지, 직전 주문과 며칠 간격인지, 누적 구매액"
WITH order_rev AS (
    SELECT
        o.user_id, o.order_id, o.order_date,
        SUM(oi.quantity * oi.unit_price) AS 주문액
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    WHERE o.status = 'completed'
    GROUP BY o.user_id, o.order_id, o.order_date
)
SELECT
    user_id,
    order_date,
    주문액,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date) AS 몇번째주문,
    order_date - LAG(order_date) OVER (PARTITION BY user_id ORDER BY order_date) AS 직전주문간격_일,
    SUM(주문액) OVER (PARTITION BY user_id ORDER BY order_date) AS 누적구매액
FROM order_rev
ORDER BY user_id, order_date;
```
