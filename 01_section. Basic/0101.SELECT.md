# Phase 1-1: 데이터 조회의 기본(SELECT)

> **목표**: `growth_shop` DB에서 원하는 데이터를 **선택·필터링·정렬**해서 뽑아낸다. SQL의 70%는 이 SELECT 문법에서 나온다.
> **선수 조건**: Phase 0 완료 (5개 테이블 + 샘플 데이터 적재됨)
> **SQLD 연계**: 2과목 SQL 기본 — SELECT 문, WHERE 절, 연산자


## 1. SELECT 기본 구조

### 1-1. 전체 조회와 컬럼 선택
```sql
-- 모든 컬럼 조회 (*는 "전체 컬럼"을 의미)
SELECT * FROM users;

-- 필요한 컬럼만 선택 (실무에서는 * 보다 명시적 선택을 권장)
SELECT name, email, channel
FROM users;
```

> 💡 PM 관점
> `SELECT *`는 빠르게 데이터를 훑을 때만 사용
> 분석 쿼리에서는 필요한 컬럼만 골라야 가독성, 성능, 모두 좋음
> BigQuery는 `SELECT *`가 스캔 비용으로 직결되니 더더욱 주의!!!

### 1-2. 별칭(Alias) - 컬럼, 테이블에 이름 붙이기
```sql
-- 컬럼 별칭: AS로 결과 컬럼명을 보기 좋게 변경
SELECT
    name        AS 회원명,
    channel     AS 획득채널,
    signup_data AS 가입일
FROM users;

-- 계산 결과에 별칭을 붙이면 의미가 명확해진다.
SELECT
    name,
    price,
    price - cost AS margin  -- 마진 = 판매가 - 원가
FROM products;
```

### 1-3. SQL 작성 순서 vs. 실행 순서 (중요)
```sql
-- 우리가 "작성"하는 순서
SELECT       컬럼           -- 5번째로 실행
FROM         테이블         -- 1번째로 실행
WHERE        조건           -- 2번째로 실행
GROUP BY     그룹           -- 3번째로 실행
HAVING       조건           -- 4번쨰로 실행
ORDER BY     정렬           -- 6번째로 실행
LIMIT        개수;          -- 7번째로 실행
```

> 💡 실행 순서
> FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT 순
> `WHERE`에서는 `SELECT`에서 만든 별칭을 쓸 수 없다. (아직 만들기 전이라서)


## 2. WHERE - 조건으로 행 걸러내기

### 2-1. 비교 연산자

| `=` | 같음 | `channel = 'paid'` |
| `<>` 또는 `!=` | 다름 | `status <> 'cancelled'` |
| `>` `<` `>=` `<=` | 대소 비교 | `price >= 50000` |

```sql
-- paid 채널로 가입한 회원만
SELECT name, channel FROM users
WHERE channel = 'paid';

-- 판매가 5만원 이상인 상품
SELECT name, price FROM products
WHERE price >= 50000;

-- 취소되지 않은 주문
SELECT order_id, status FROM orders
WHERE status <> 'cancelled';
```

### 2-2. 논리 연산자 (AND/OR/NOT)
```sql
-- AND: 두 조건 모두 만족 (한국에서 paid로 가입한 회원)
SELECT name FROM users
WHERE country = 'KR' AND channel = 'paid';

-- OR: 둘 중 하나라도 만족 (organic 또는 referral)
SELECT name, channel FROM users
WHERE channel = 'organic' OR channel = 'referral';

-- NOT: 조건 부정
SELECT name FROM users
WHERE NOT country = 'KR';   -- 한국이 아닌 회원

-- 괄호로 우선순위 명시 (AND가 OR보다 먼저 실행됨!)
SELECT name FROM users
WHERE country = 'KR'
  AND (channel = 'paid' OR channel = 'referral');
```

> ⚠️ AND가 OR보다 우선순위가 높다.
> 의도와 다른 결과를 막으려면 괄호로 묶는 습관이 안전하다.


## 3. 범위·목록·패턴 필터링

### 3-1. BETWEEN - 범위
```sql
-- 가격이 2만원 ~ 7만원 사이 (양 끝값 포함)
SELECT name, price FROM products
WHERE price BETWEEN 20000 AND 70000;

-- 날짜 범위 (1~2월 가입자) — 코호트 분석의 출발점
SELECT name, signup_date FROM users
WHERE signup_date BETWEEN '2024-01-01' AND '2024-02-29';
```

> `BETWEEN a AND b`는 `>= a AND <= b`와 같다 (경계값 포함)

### 3-2. IN - 목록
```sql
-- 여러 카테고리를 한 번에 (OR를 줄여줌)
SELECT name, category FROM products
WHERE category IN ('electronics', 'lifestyle');

/*
WHERE category = 'electronics'
   OR category = 'lifestyle';
*/

-- NOT IN: 목록 제외
SELECT order_id, status FROM orders
WHERE status NOT IN ('cancelled', 'refunded');  -- 유효 주문만

/*
WHERE status != 'cancelled'
  AND status != 'refunded';
*/

```

### 3-3. LIKE - 패턴 매칭 (와일드 카드)

| 와일드카드 | 의미 |
|-----------|------|
| `%` | 0개 이상의 임의 문자 |
| `_` | 정확히 1개의 임의 문자 |

```sql
-- 이메일이 test.com으로 끝나는 회원
SELECT email FROM users
WHERE email LIKE '%test.com';

-- 이름이 '김'으로 시작하는 회원
SELECT name FROM users
WHERE name LIKE '김%';

-- 대소문자 무시 검색 (PostgreSQL 전용: ILIKE)
SELECT name FROM products
WHERE name ILIKE '%usb%';   -- USB, usb 모두 매칭
```

> 💡 `ILIKE`는 PostgreSQL 고유 기능(대소문자 무시).
> MySQL·BigQuery엔 없으니 표준이 필요하면 `LOWER(컬럼) LIKE '%usb%'`를 쓴다.


## 4. NULL 다루기 (가장 흔한 실수)
`NULL`은 "값이 없음/알 수 없음"이며, 0이나 빈 문자열과 다르다.
그래서 `= NULL`은 절대 동작하지 않음.

```sql
-- ❌ 틀림: NULL은 = 로 비교 불가 (결과가 항상 '알 수 없음')
SELECT name FROM products WHERE cost = NULL;

-- ✅ 올바름: IS NULL / IS NOT NULL 사용
SELECT name FROM products WHERE cost IS NULL;       -- 원가가 없는 상품
SELECT name FROM users   WHERE channel IS NOT NULL; -- 채널값이 있는 회원
```

> 💡 PM 관점
> 데이터 품질 점검의 기본이 IS NULL 카운트
> "채널값이 비어있는 회원이 몇 명인지"를 보는 게 분석 신뢰도의 첫걸음.


## 5. 정렬, 제한, 중복 제거

### 5-1. ORDER BY - 정렬
```sql
-- 가격 높은 순 (DESC = 내림차순)
SELECT name, price FROM products
ORDER BY price DESC;

-- 가격 낮은 순 (ASC = 오름차순, 기본값이라 생략 가능)
SELECT name, price FROM products
ORDER BY price ASC;

-- 다중 정렬: 채널 오름차순 → 같으면 가입일 최신순
SELECT name, chanel, signup_date FROM users
ORDER BY channel ASC, signup_date DESC;
```

> `ORDER BY`에서는 `SELECT`의 별칭을 쓸 수 있음. (실행 순서상 SELECT 다음이라서)
> `ORDER BY 1, 2`처럼 컬럼 순번으로도 정렬 가능

### 5-2. LIMIT / OFFSET - 개수 제한 & 페이징
```sql
-- 가장 비싼 상품 TOP 3
SELECT name, price FROM products
ORDER BY price DESC
LIMIT 3;

-- 페이징: 4~6번째 (앞 3개 건너뛰고 3개)
SELECT name, price FROM products
ORDER BY price DESC
LIMIT 3 OFFSET 3;
```

> 💡 "TOP N 상품/회원" 같은 랭킹은 실무 최빈 패턴.
> 더 정교한 순위 매기기는 Phase 6(윈도우 함수)에서 RANK()로 발전시킨다.

### 5-3. DISTINCT - 중복 제거
```sql
-- 우리 회원의 획득 채널 종류 (중복 제거)
SELECT DISTINCT channel FROM users

-- 여러 컬럼 조합의 고유값 (국가+채널 조합)
SELECT DISTINCT country, channel FROM users
ORDER BY country, channel;
```

> ⚠️ `DISTINCT`는 **모든 선택 컬럼의 조합**에 적용됨.
> `SELECT DISTINCT a, b`는 "a의 중복 제거"가 아니라 "(a,b) 쌍의 중복 제거"다.


### 6. 종합 예시
```sql
-- "2026년 1~2월에 paid/referral 채널로 가입한 한국 회원을,
--   가입일 최신순으로 최대 5명 조회"
SELECT
    name        AS 회원면,
    channel     AS 채널,
    signup_date AS 가입일
FROM users
WHERE country = 'KR'
  AND channel IN ('paid', 'referral')
  AND signup_date BETWEEN '2026-01-01' AND '2026-02-29'
ORDER BY signup_date DESC
LIMIT 5;
```

### 🟡 Lv.2-B — LIKE 다중 + 부정 채널

문제. 이름이 **'김' 또는 '이'로 시작**하는 회원 중, 채널이 **paid가 아닌** 회원을 가입일 **빠른 순**으로 조회하시오. (컬럼: 회원명·채널·가입일)

```sql
SELECT name, channel, signup_date FROM users
WHERE (name LIKE '김%' OR name LIKE '이%')
  AND channel <> 'paid'
ORDER BY signup_date ASC;

-- `LIKE`는 와일드 카드 사용 가능, `IN`은 일치하는 것만 가능
```

### 🔴 Lv.3-A — [분석 미션] 환불/취소 주문 모니터링

상황. CS팀이 요청했다.
> "정상 완료(`completed`)가 **아닌** 주문을 모니터링하고 싶어요.
> 취소·환불 주문의 `order_id`, 회원 ID, 주문일, 상태를 **주문일 최신순**으로 보여주세요.
> 결과 컬럼명은 한글로요."

- 요구사항
1. 단일 SELECT 쿼리 작성 (별칭 한글화)
2. `complete`가 아닌 조건을 두 가지 방법 (`<>`사용 / `IN` 사용)으로 각각 작성하고, 둘의 결과가 같은지 비교
3. `phase1_refund_monitor.sql`로 저장 + 주석에 비즈니스 질문 기록 + Github 커밋

```sql
-- 방법 1. 부등호 사용
SELECT
    order_id    AS 주문번호,
    user_id     AS 회원ID,
    order_date  AS 주문일,
    status      AS 상태    
FROM orders
WHERE status <> 'completed'
ORDER BY order_date DESC;
-- 미래에 `pending`, `shipping` 등의 상태가 생기면 자동 포함

-- 방법 2. IN 사용 (상태값이 명확히 정해져 있을 때 더 명시적)
SELECT order_id, user_id, order_date, status
FROM orders
WHERE status IN ('cancelled', 'refunded')
ORDER BY order_date DESC;
-- 명시한 값만 포함
```

### 🔴 Lv.3-B — [분석 미션] 프리미엄 상품 진열 후보
