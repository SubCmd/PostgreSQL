## 1. 인덱스 (Index)

인덱스는 책의 "찾아보기"처럼, 특정 컬럼 값을 빠르게 찾도록 미리 정렬해둔 자료구조다.

```sql
-- 기본 B-tree 인덱스 (=, <, >, BETWEEN, ORDER BY에 효과)
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_date ON orders(order_date);

-- 복합 인덱스 (여러 컬럼) — 컬럼 순서가 중요!
CREATE INDEX idx_orders_user_date ON orders(user_id, order_date);

-- 부분 인덱스 (특정 조건 행만 인덱싱 → 작고 빠름)
CREATE INDEX idx_orders_completed ON orders(order_date)
WHERE status = 'completed';

-- 인덱스 확인 / 삭제
\d orders                       -- (psql) 인덱스 목록 포함
DROP INDEX IF EXISTS idx_orders_date;
```

### 인덱스 종류

| 종류 | 적합한 경우 |
|------|------------|
| **B-tree** (기본) | 등호·범위·정렬 (`=`, `<`, `BETWEEN`, `ORDER BY`) |
| **GIN** | `JSONB`, 배열, 전문검색(다중 값 포함) |
| **GiST** | 범위·기하·근접 검색 |
| **Hash** | 등호(`=`)만 (거의 B-tree로 충분) |


### 복합 인덱스의 "왼쪽 접두" 규칙 ⚠️

```sql
-- idx_orders_user_date = (user_id, order_date)
-- ✅ 활용됨: WHERE user_id = 1
-- ✅ 활용됨: WHERE user_id = 1 AND order_date > '...'
-- ❌ 활용 안 됨: WHERE order_date > '...'  (선행 컬럼 user_id 조건 없음)
```

> 💡 복합 인덱스는 **선행 컬럼부터** 써야 한다.
> 자주 같이 쓰는 컬럼을 "선택도 높은 것(값이 다양한 것) → 낮은 것" 순으로 배치.


### 인덱스의 비용 (공짜가 아니다)

> ⚠️ 인덱스는 조회는 빠르게 하지만, `INSERT/UPDATE/DELETE` 시 인덱스도 갱신해야 해 **쓰기는 느려지고 저장공간도 쓴다**.
> "모든 컬럼에 인덱스"는 안티패턴. **자주 필터·조인·정렬하는 컬럼**에 선별적으로.

## 2. EXPLAIN / EXPLAIN ANALYZE — 실행계획 읽기

옵티마이저가 쿼리를 **어떻게 실행할지** 보여준다. 느린 쿼리 진단의 출발점.

```sql
-- EXPLAIN: 예상 계획만 (실행 안 함)
EXPLAIN
SELECT * FROM orders WHERE user_id = 1;
 
-- EXPLAIN ANALYZE: 실제 실행 + 실측 시간/행수
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 1;
```


### 읽는 법 (핵심 키워드)

| 노드 | 의미 |
|------|------|
| **Seq Scan** | 테이블 전체 훑기 (인덱스 미사용) — 큰 테이블이면 느림 |
| **Index Scan** | 인덱스로 바로 찾기 — 빠름 |
| **Bitmap Heap Scan** | 인덱스로 여러 행 후보 모은 뒤 테이블 접근 |
| **Nested Loop / Hash Join / Merge Join** | 조인 방식 |
| `cost=0.00..35.50` | 예상 비용(시작..총) |
| `actual time=0.02..0.05 rows=3` | 실측 시간(ms)·행수 |
 
```sql
-- 인덱스 효과 체험: 인덱스 만들기 전후로 EXPLAIN ANALYZE 비교
-- (작은 연습 DB에선 Seq Scan이 더 쌀 수 있음 → 데이터가 많아야 Index Scan 채택)
```
 
> 💡 **PM/분석가 관점**: "이 대시보드 쿼리가 왜 느리지?" → `EXPLAIN ANALYZE`로 Seq Scan이 큰 테이블에 걸렸는지 확인 → 해당 필터/조인 컬럼에 인덱스. 추정 행수와 실제 행수가 크게 다르면 통계가 오래된 것(`ANALYZE 테이블;`로 갱신).

## 3. 뷰(VIEW) & 머티리얼라이즈드 뷰 (Materialized View)

### 3-1. 일반 뷰 - 저장된 쿼리 (가상 테이블)

```sql
-- 자주 쓰는 분석 쿼리를 뷰로 저장 → 이름으로 재사용
CREATE VIEW v_completed_revenue AS
SELECT o.user_id, o.order_id, o.order_date,
       SUM(oi.quantity * oi.unit_price) AS 매출
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.status = 'completed'
GROUP BY o.user_id, o.order_id, o.order_date;

-- 일반 테이블처럼 조회
SELECT user_id, SUM(매출) FROM v_completed_revenue GROUP BY user_id;
```

> 뷰는 **데이터를 저장하지 않는다**.
> 조회할 때마다 원본 쿼리가 실행된다 → 항상 최신, 단 무거운 쿼리면 매번 느림.


### 3-2. 머티리얼라이즈드 뷰 — 결과를 저장

```sql
-- 결과를 '물리적으로' 저장 (무거운 집계를 미리 계산)
CREATE MATERIALIZED VIEW mv_daily_sales AS
SELECT o.order_date AS 날짜,
       SUM(oi.quantity * oi.unit_price) AS 매출
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.status = 'completed'
GROUP BY o.order_date;

-- 조회는 빠름(저장된 결과). 단, 원본이 바뀌면 수동 갱신 필요
REFRESH MATERIALIZED VIEW mv_daily_sales;
```

| | VIEW | MATERIALIZED VIEW |
|--|------|-------------------|
| 데이터 저장 | X (매번 실행) | O (결과 저장) |
| 최신성 | 항상 최신 | `REFRESH` 전까지 과거 |
| 속도 | 원본 쿼리 속도 | 빠름(저장됨) |
| 용도 | 가벼운 추상화 | 무거운 집계 대시보드 |
 
> 💡 Phase 7의 "일별 집계 테이블 + UPSERT"를 MV로 대체할 수 있다.
> 실무에선 MV·집계테이블·dbt 모델 중 갱신 주기/제어 수준에 따라 선택.
