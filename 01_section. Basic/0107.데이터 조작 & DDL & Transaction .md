## 1. INSERT - 데이터 추가
```sql
-- 단건 삽입 (컬럼 명시 권장)
INSERT INTO products (name, category, price, cost)
VALUES ('마우스패드', 'lifestyle', 12000, 4000);

-- 다건 삽입 (한 번에)
INSERT INTO products (name, category, price, cost)
VALUES ('웹캠', 'electronics', 55000, 28000),
       ('스탠드', 'lifestyle', 38000, 15000);

-- INSERT ... SELECT : 쿼리 결과를 그대로 삽입 (집계 결과 적재 등)
INSERT INTO products (name, category, price, cost)
SELECT name || ' (리퍼)', category, ROUND(price * 0.7), cost
FROM products
WHERE category = 'electronics';

-- RETURNING : 삽입된 행(자동 생성 PK 등)을 즉시 반환
INSERT INTO users (email, name, signup_date, channel)
VALUES ('new@test.com', '신규회원', CURRENT_DATE, 'organic')
RETURNING user_id, email;
-- 특정 컬럼만 콕 집어서 확인해주는 원리
-- 물건 구입하고 영수증 바로 발급 받는 느낌.
```

> 💡 `SERIAL` PK는 자동 생성되므로 INSERT에서 생략한다.
> `RETURNING`은 PostgreSQL의 강력한 기능 — 방금 생성된 ID를 애플리케이션이 바로 쓸 수 있어 라운드트립을 줄인다.

## 2. UPDATE - 데이터 수정

```sql
-- ⚠️ WHERE 없는 UPDATE는 '전체 행'을 바꾼다. 항상 WHERE 확인!
UPDATE products
SET price = price * 1.1             -- 10% 인상
WHERE category = 'electronics';

-- 여러 컬럼 동시 수정
UPDATE users
SET channel = 'paid', country = 'KR'
WHERE user_id = 1;

-- JOIN 기반 UPDATE (다른 테이블 값 참조) - PostgreSQL은 FROM 사용
UPDATE order_items oi
SET unit_price = p.price                -- 현재 상품가로 갱신(예시)
FROM products p
WHERE oi.product_id = p.product_id
  AND oi.order_id = 999;                -- 특정 주문만
```

> ⚠️ **가장 위험한 실수**
> `WHERE` 없는 `UPDATE`/`DELETE`.
> 실무에선 먼저 같은 조건으로 `SELECT`해 영향 행을 확인한 뒤 실행하는 습관이 필수.

## 3. DELETE - 데이터 삭제

```sql
-- 조건부 삭제
DELETE FROM products
WHERE name LIKE '%(리퍼)';

-- 서브쿼리 조건 삭제 (주문이 한 번도 없는 임시 데이터 정리 등)
DELETE FROM products
WHERE product_id NOT IN (SELECT DISINTCT product_id FROM order_items);

-- TRUNCATE : 테이블 전체를 빠르게 비움 (롤백 주의, 개별 WHERE 불가)
-- TRUNCATE TABLE 임시 테이블;
```

| | DELETE | TRUNCATE |
|--|--------|----------|
| 조건(WHERE) | 가능 | 불가(전체) |
| 속도 | 행마다 처리(느림) | 매우 빠름 |
| 트랜잭션 롤백 | 가능 | (PostgreSQL은 가능) |

## 4. UPSERT - 있으면 수정, 없으면 삽입 (ON CONFLICT)

```sql
-- email이 UNIQUE라고 가정. 이미 있으면 이름 갱신, 없으면 새로 삽입
INSERT INTO users (email, name, signup_date, channel)
VALUES ('kim@test.com', '김민준(수정)', CURRENT_DATE, 'organic')
ON CONFLICT (email)
DO UPDATE SET name = EXCLUDED.name;     -- EXCLUDED = 삽입하려던 값

-- 충돌 시 아무것도 안 함(중복 무시)
INSERT INTO users (email, name, signup_date)
VALUES ('kim@test.com', '중복', CURRENT_DATE)
ON CONFLICT (email) DO NOTHING;
```

> 💡 `UPSERT`는 "일별 지표 테이블을 매일 갱신(있으면 업데이트, 없으면 삽입)" 같은 적재 파이프라인에서 핵심. 
> `EXCLUDED`는 삽입하려다 충돌한 그 값을 가리킨다.

## 5. DDL - 테이블 구조 정의

### 5-1. CREATE / ALTER / DROP

```sql
CREATE TABLE coupons (
    coupon_id   SERIAL PRIMARY KEY,
    code        VARCHAR(50) UNIQUE NOT NULL,
    discount    NUMERIC(10, 2) NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW()
)

-- 컬럼 추가 / 변경 / 삭제
ALTER TABLE users ADD COLUMN grade VARCHAR(20) DEFULAT  'bronze';
ALTER TABLE users ALTER COLUMN grade SET NOT NULL;
ALTER TABLE users RENAME COLUMN grade TO membership;
ALTER TABLE users DROP COLUMN grade membership;

-- 테이블 삭제 (IF EXISTS로 안전하게)
DROP TABLE IF EXISTS coupons;
```

### 5-2. 자주 쓰는 데이터 타입

| 분류 | 타입 | 용도 |
|------|------|------|
| 정수 | `INTEGER`, `BIGINT`, `SERIAL` | ID, 카운트 |
| 실수/금액 | `NUMERIC(p,s)`, `REAL` | 가격(정확도 위해 NUMERIC) |
| 문자 | `VARCHAR(n)`, `TEXT` | 이름, 본문 |
| 날짜/시간 | `DATE`, `TIMESTAMP`, `TIMESTAMPTZ` | 가입일, 이벤트 시각 |
| 불리언 | `BOOLEAN` | 플래그 |
| 반구조 | `JSONB`, `ARRAY` | 속성, 태그(Phase 8) |
 
> ⚠️ 금액은 `FLOAT/REAL`이 아니라 **`NUMERIC`**.
> 부동소수점 오차로 0.1+0.2≠0.3 문제가 돈 계산에선 치명적.
