# Phase 1-7: 데이터 조작 & DDL & 트랜젝션

> **목표**: 읽기에서 **쓰기**로. `INSERT/UPDATE/DELETE`로 데이터를 변경하고, `UPSERT`·제약조건·트랜잭션(ACID)으로 **안전하게** 다룬다. "분석가"에서 "데이터를 신뢰성 있게 운용하는 사람"으로 확장.
> **선수 조건**: Phase 1~6 완료
> **SQLD 연계**: 2과목 — DML, DDL, TCL(트랜잭션 제어), 제약조건
 
> 🛡️ **실습 안전 수칙**: 이 페이즈는 데이터를 **변경**한다. 연습 DB(`growth_shop`)의 기존 데이터를 보존하려면, 변경 실습은 **트랜잭션으로 감싸고 `ROLLBACK`** 하거나, **스크래치 테이블**에서 연습해야 함. (9장 트랜잭션 참고)

## 1. INSERT - 데이터 추가
```sql
-- 단건 삽입 (컬럼 명시 권장)
INSERT INTO products (name, category, price, cost)
VALUES ('마우스패드', 'lifestyle', 12000, 4000);

-- 다건 삽입 (한 번에)
INSERT INTO products (name, category, price, cost)
VALUES ('웹캠', 'electronics', 55000, 28000),
       ('스탠드', 'lifestyle', 38000, 15000);

-- INSERT ... SELECT : 쿼리 결과를 그대로 삽입 (집계 결과 적재 등)
INSERT INTO products (name, category, price, cost)
SELECT name || ' (리퍼)', category, ROUND(price 확정안내문자 X

- 생기부 길라잡이 X
- 중간, 기말 성적표 X* 0.7), cost
FROM products
WHERE category = 'electronics';

-- RETURNING : 삽입된 행(자동 생성 PK 등)을 즉시 반환
INSERT INTO users (email, name, signup_date, channel)
VALUES ('new@test.com', '신규회원', CURRENT_DATE, 'organic')
RETURNING user_id, email;
-- 특정 컬럼만 콕 집어서 확인해주는 원리
-- 물건 구입하고 영수증 바로 발급 받는 느낌.
```

> 💡 `SERIAL` PK는 자동 생성되므로 INSERT에서 생략한다.
> `RETURNING`은 PostgreSQL의 강력한 기능 — 방금 생성된 ID를 애플리케이션이 바로 쓸 수 있어 라운드트립을 줄인다.

## 2. UPDATE - 데이터 수정

```sql
-- ⚠️ WHERE 없는 UPDATE는 '전체 행'을 바꾼다. 항상 WHERE 확인!
UPDATE products
SET price = price * 1.1             -- 10% 인상
WHERE category = 'electronics';

-- 여러 컬럼 동시 수정
UPDATE users
SET channel = 'paid', country = 'KR'
WHERE user_id = 1;

-- JOIN 기반 UPDATE (다른 테이블 값 참조) - PostgreSQL은 FROM 사용
UPDATE order_items oi
SET unit_price = p.price                -- 현재 상품가로 갱신(예시)
FROM products p
WHERE oi.product_id = p.product_id
  AND oi.order_id = 999;                -- 특정 주문만
```

> ⚠️ **가장 위험한 실수**
> `WHERE` 없는 `UPDATE`/`DELETE`.
> 실무에선 먼저 같은 조건으로 `SELECT`해 영향 행을 확인한 뒤 실행하는 습관이 필수.

## 3. DELETE - 데이터 삭제

```sql
-- 조건부 삭제
DELETE FROM products
WHERE name LIKE '%(리퍼)';

-- 서브쿼리 조건 삭제 (주문이 한 번도 없는 임시 데이터 정리 등)
DELETE FROM products
WHERE product_id NOT IN (SELECT DISINTCT product_id FROM order_items);

-- TRUNCATE : 테이블 전체를 빠르게 비움 (롤백 주의, 개별 WHERE 불가)
-- TRUNCATE TABLE 임시 테이블;
```

| | DELETE | TRUNCATE |
|--|--------|----------|
| 조건(WHERE) | 가능 | 불가(전체) |
| 속도 | 행마다 처리(느림) | 매우 빠름 |
| 트랜잭션 롤백 | 가능 | (PostgreSQL은 가능) |

## 4. UPSERT - 있으면 수정, 없으면 삽입 (ON CONFLICT)

```sql
-- email이 UNIQUE라고 가정. 이미 있으면 이름 갱신, 없으면 새로 삽입
INSERT INTO users (email, name, signup_date, channel)
VALUES ('kim@test.com', '김민준(수정)', CURRENT_DATE, 'organic')
ON CONFLICT (email)
DO UPDATE SET name = EXCLUDED.name;     -- EXCLUDED = 삽입하려던 값

-- 충돌 시 아무것도 안 함(중복 무시)
INSERT INTO users (email, name, signup_date)
VALUES ('kim@test.com', '중복', CURRENT_DATE)
ON CONFLICT (email) DO NOTHING;
```

> 💡 `UPSERT`는 "일별 지표 테이블을 매일 갱신(있으면 업데이트, 없으면 삽입)" 같은 적재 파이프라인에서 핵심. 
> `EXCLUDED`는 삽입하려다 충돌한 그 값을 가리킨다.

## 5. DDL - 테이블 구조 정의

### 5-1. CREATE / ALTER / DROP

```sql
CREATE TABLE coupons (
    coupon_id   SERIAL PRIMARY KEY,
    code        VARCHAR(50) UNIQUE NOT NULL,
    discount    NUMERIC(10, 2) NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW()
)

-- 컬럼 추가 / 변경 / 삭제
ALTER TABLE users ADD COLUMN grade VARCHAR(20) DEFULAT  'bronze';
ALTER TABLE users ALTER COLUMN grade SET NOT NULL;
ALTER TABLE users RENAME COLUMN grade TO membership;
ALTER TABLE users DROP COLUMN grade membership;

-- 테이블 삭제 (IF EXISTS로 안전하게)
DROP TABLE IF EXISTS coupons;
```

### 5-2. 자주 쓰는 데이터 타입

| 분류 | 타입 | 용도 |
|------|------|------|
| 정수 | `INTEGER`, `BIGINT`, `SERIAL` | ID, 카운트 |
| 실수/금액 | `NUMERIC(p,s)`, `REAL` | 가격(정확도 위해 NUMERIC) |
| 문자 | `VARCHAR(n)`, `TEXT` | 이름, 본문 |
| 날짜/시간 | `DATE`, `TIMESTAMP`, `TIMESTAMPTZ` | 가입일, 이벤트 시각 |
| 불리언 | `BOOLEAN` | 플래그 |
| 반구조 | `JSONB`, `ARRAY` | 속성, 태그(Phase 8) |
 
> ⚠️ 금액은 `FLOAT/REAL`이 아니라 **`NUMERIC`**.
> 부동소수점 오차로 0.1+0.2≠0.3 문제가 돈 계산에선 치명적.

## 6. 제약조건 (Constraints)

```sql
CREATE TABLE reviews (
    review_id  SERIAL PRIMARY KEY,                    -- PK: 유일 식별
    user_id    INTEGER NOT NULL,                      -- NOT NULL: 필수
    product_id INTEGER NOT NULL,
    rating     SMALLINT CHECK (rating BETWEEN 1 AND 5),-- CHECK: 값 범위 제한
    content    TEXT,
    created_at TIMESTAMP DEFAULT NOW(),               -- DEFAULT: 기본값
    UNIQUE (user_id, product_id),                     -- UNIQUE: 회원당 상품 1리뷰
    FOREIGN KEY (user_id) REFERENCES users(user_id)
        ON DELETE CASCADE,                            -- 회원 삭제 시 리뷰도 삭제
    FOREIGN KEY (product_id) REFERENCES products(product_id)
        ON DELETE RESTRICT                            -- 참조 중이면 상품 삭제 막음
);
```

### FK의 ON DELETE 옵션
 
| 옵션 | 부모 삭제 시 자식 |
|------|------------------|
| `RESTRICT`/`NO ACTION` | 삭제 막음(기본) |
| `CASCADE` | 함께 삭제 |
| `SET NULL` | FK를 NULL로 |
 
> 💡 제약조건은 **데이터 무결성을 DB가 강제**하게 한다.
> 애플리케이션 코드 버그가 있어도 잘못된 데이터가 들어가지 못하게 막는 안전망.

## 7. 트랜젝션 (Transaction) & ACID ⭐
여러 변경을 하나의 단위로 묶여, 전부 성공하거나 전부 취소되게 함.

```sql
BEGIN;                                    -- 트랜잭션 시작

UPDATE accounts SET balance = balance - 1000 WHERE id = 1;  -- 출금
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;  -- 입금

-- 둘 다 성공하면:
COMMIT;                                   -- 확정
-- 문제가 생기면:
-- ROLLBACK;                              -- 전부 취소(이전 상태로)
```

### ACID (트랜젝션의 4대 보장)

| 속성 | 의미 |
| ---- | ---- | 
| Atomicity 원자성 | 전부 성공 or 전부 실패 (중간 없음) |
| Consistency 일관성 | 제약조건을 항상 만족하는 상태 유지 | 
| Isolation 격리성 | 동시 트랜젝션이 서로 간섭 안 함 |
| Durability 지속성 | 커밋된 변경은 장애에도 보존 |


### SAVEPOINT (부분 롤백)

```sql
BEGIN;
INSERT INTO products (name, category, price) VALUES ('A', 'x', 100);
SAVEPOINT sp1;
INSERT INTO products (name, category, price) VALUES ('B', 'x', 200);
ROLLBACK TO sp1;    -- B만 취소, A는 유지
COMMIT;             -- A만 확정
```

> 💡 **학습 활용**
> 변경 실습을 `BEGIN; ... ROLLBACK;`으로 감싸면 결과를 확인하되 DB는 원상복구된다.
> 연습 DB를 보호하는 안전한 방법.

## 8. 정규화 맛보기 (Phase 8 예고)
데이터 중복·이상(anomaly)을 줄이려 테이블을 나누는 게 정규화

- 1NF: 한 칸에 하나의 값 (배열·반복 그룹 제거)
- 2NF: 부분 함수 종속 제거 (복합키 일부에만 의존하는 컬럼 분리)
- 3NF: 이행 함수 종속 제거 (키가 아닌 컬럼에 의존하는 컬럼 분리)

> 우리 스키마가 `orders`/`order_items`/`products`로 나뉜게 정규화 결과
> 자세한 1NF~3NF·BCNF·반정규화는 Phase 8에서. SQLD 1과목 핵심!

## 9. 종합 예시 (실무형) - 안전한 가격 일괄 인상

```sql
BEGIN;

-- 1) 영향 범위 미리 확인 (변경 전 SELECT 습관)
SELECT product_id, name, price, ROUND(price*1.1) AS 인상후
FROM products WHERE category = 'electronics';

-- 2) 변경 실행 + 결과 반환
UPDATE products
SET price = ROUND(price*1.1)
WHERE category = 'electronics'
RETURNING product_id, name, price;

-- 3) 확인 후 확정 또는 취소
-- COMMIT;      -- 문제 없으면 확정
ROLLBACK;       -- 학습 중엔 롤백으로 원상복구
```

## 10. 연습 문제

**Q1.** `products`에 새 상품 1건('노트북 거치대', lifestyle, 28000, 11000)을 삽입하시오.

```sql
UPDATE users

WHERE user_id = 8;
```

**Q3.** 가격이 10000원 미만인 상품을 삭제하는 쿼리를 작성하시오. (조건 확인 먼저)

```sql
-- 먼저 확인
SELECT * FROM products WHERE price < 10000;

-- 삭제
DELETE FROM products WHERE price < 10000;
```

### 🟡 Lv.2 — 응용
 
**Q4.** `users.email`이 UNIQUE라고 할 때, `'kim@test.com'`이 있으면 이름을 '김민준(VIP)'로 갱신하고, 없으면 새로 삽입하는 `UPSERT`를 작성하시오.
```sql
INSERT INTO users (email, name, signup_date, channel)
VALUES ('kim@test.com', '김민준(VIP)', CURRENT_DATE, 'organic')
ON CONFLICT (email)
DO UPDATE SET name = EXCLUDED.name;
```

**Q5.** `electronics` 카테고리 상품의 가격을 5% 인상하되, 변경된 행의 상품명·새 가격을 `RETURNING`으로 반환하시오. (트랜잭션으로 감싸고 롤백)
```sql
BEGIN;
UPDATE products;
SET price = ROUND(price*1.05)
WHERE category = 'electronics'
RETURNING name, price;
ROLLBACK;       -- 학습용: 확인 후 원상복구
```

**Q6.** 별점(1~5)을 강제하는 `CHECK` 제약과, '회원당 상품 1리뷰'를 강제하는 `UNIQUE` 제약, 회원 삭제 시 리뷰도 삭제되는 FK(`ON DELETE CASCADE`)를 가진 `reviews` 테이블을 생성하시오.
```sql
CREATE TABLE reviews (
    review_id  SERIAL PRIMARY KEY,
    user_id    INTEGER NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    rating     SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
    content    TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE (user_id, product_id)
);
```
