# Phase 1-4: 조인 (JOIN)

> **목표**: 여러 테이블을 **연결**해 진짜 비즈니스 지표를 뽑는다. "회원별 총 구매액", "카테고리별 매출", "한 번도 주문 안 한 회원" 같은 분석이 가능해진다. 실무·SQLD 최빈출 구간.
> **선수 조건**: Phase 1~3 완료 (`GROUP BY`, 조건부 집계 자유 구사)
> **SQLD 연계**: 2과목 — 조인의 종류, 표준 조인, 카티션 곱


## 0. 이 페이지에서 얻어갈 것

- [ ] `INNER JOIN`과 `LEFT JOIN`의 차이를 그림으로 설명
- [ ] `ON`(조인 조건)과 `WHERE`(필터)의 역할 구분
- [ ] JOIN + `GROUP BY`로 회원별/카테고리별 지표 산출
- [ ] `LEFT JOIN ... IS NULL`로 "없는 것" 찾기
- [ ] SELF JOIN / CROSS JOIN의 쓰임
- [ ] **JOIN 함정**: 행 증식(fan-out), LEFT JOIN에서 WHERE의 함정

## 1. JOIN이 왜 필요한가

Phase 0에서 데이터를 정규화해 여러 테이블로 나눔. (회원·주문·상품)
"회원 이름 + 그 회원의 주문 매출"을 함께 보려면 테이블을 다시 연결해야 함.
연결 도구가 [JOIN]

```
users          orders              order_items         products
user_id 🔑 ──< user_id 🔗         order_id 🔗   >──  order_id 🔑
               order_id 🔑 ──< ... product_id 🔗 ──>  product_id 🔑
```

연결고리(키)로 테이블을 이음.
`users.users_id = orders.user_id`, `orders.order_id = order_items.order_id`,
`order_items.product_id = products.product_id`

## 2. INNER JOIN - 양쪽 모두 있는 것만

```sql
-- 주문한 회원의 이름과 주문 정보 (양쪽에 매칭되는 행만)
SELECT
    u.name       AS 회원명,
    o.order_id   AS 주문번호,
    o.order_date AS 주문일,
    o.status     AS 상태
FROM users AS u
INNER JOIN orders AS o
        ON u.user_id = o.user_id        -- 연결 조건
ORDER BY o.order_date;
```

> 💡 **별칭(alias) 습관**
> 테이블에 `u`, `o` 같은 별칭을 붙이고 컬럼 앞에 `u.`, `o.`를 명시함.
> 여러 테이블에 같은 컬럼명(`user_id`)이 있을 때 모호함을 없애고 가독성도 좋음.
 
`INNER JOIN`은 **양쪽 테이블에서 조인 조건이 맞는 행만** 남긴다.
주문이 없는 회원은 결과에서 빠짐.

## 3. LEFT JOIN - 왼쪽은 모두 보존

```sql
-- 모든 회원 + (있으면) 주문. 주문 없는 회원도 NULL로 표시
SELECT
    u.name,
    o.order_id,
    o.status
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
ORDER BY u.user_id;
-- 주문 없는 회원(예: 임채원)은 order_id가 NULL로 나옴.
```

| 조인 | 결과 |
|------|------|
| `INNER JOIN` | 양쪽 매칭되는 행만 (교집합) |
| `LEFT JOIN` | 왼쪽 전부 + 매칭되는 오른쪽 (없으면 NULL) |
| `RIGHT JOIN` | 오른쪽 전부 + 매칭되는 왼쪽 |
| `FULL OUTER JOIN` | 양쪽 전부 (매칭 안 되면 각각 NULL) |

> 💡 **실무 팁**
> `LEFT JOIN`이 가장 많이 쓰임.
> "기준 테이블(회원/상품)은 다 보여주고, 연관 데이터를 붙인다"는 분석 패턴에 맞기 때문.
> `RIGHT JOIN`은 `LEFT`로 바꿔 쓸 수 있어 거의 사용하지 않음.

### "없는 것" 찾기 - LEFT JOIN + IS NULL

```sql
-- 한 번도 주문하지 않은 회원 (이탈/미활성 회원 발굴)
SELECT u.user_id, u.name, u.channel
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
WHERE o.order_id IS NULL;           -- 매칭 안 된 행 = 주문 없음
```

> 💡 매우 자주 쓰는 패턴
> "구매 전환 안 한 회원", "리뷰 없는 상품" 등 **부재(absence)** 를 찾을 때.

## 4. JOIN + GROUP BY - 진짜 지표의 시작 ⭐
이게 Phase 3과 Phase 4가 만나는 지점이자, 실무 분석의 핵심.

```sql
-- 회원별 주문 건수 (주문 없는 회원도 0으로 표시)
SELECT
    u.name,
    COUNT(o.orderid) AS 주문건수    -- COUNT(컬럼)은 NULL 제외 → 주문 없으면 0
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
GROUP BY u.user_id, u.name          -- PK까지 묶어야 동명이인 안전
ORDER BY 주문건수 DESC;
```

> ⚠️ `COUNT(o.order_id)`와 `COUNT(*)`의 차이
> LEFT JOIN에서 주문 없는 회원은 `o.order_id`가 NULL이라 `COUNT(o.order_id)=0`
> 하지만 `COUNT(*)`는 그 NULL 행도 1로 세어 **1**이 된다. 주문 건수를 정확히 세려면 `COUNT(컬럼)`.

```sql
-- 카테고리별 총 매출 (3개 테이블 조인)
SELECT
    p.category,
    SUM(oi.quantity * oi.unit_price) AS 총매출,
    COUNT(DISTINCT oi.order_id)      AS 주문수
FROM order_items AS oi
INNER JOIN products AS p
    ON oi.product_id = p.product_id
GROUP BY p.category
ORDER BY 총매출 DESC;
```

## 5. 다중 테이블 조인 (4개 연결)

```sql
-- "회원별 총 구매액" - 유효 주문(completed)만, 매출 높은 순
SELECT
    u.name AS 회원명,
    u.channel AS 채널,
    COUNT(DISTINCT o.order_id)       AS 주문수,
    SUM(oi.quantity * oi.unit_price) AS 총구매액
FROM users AS u
INNER JOIN orders AS o
        ON u.user_id = o.user_id
       AND o.status = 'completed'       -- 조인 시점에 상태 필터 (4장 함정 참고)
INNER JOIN order_items AS oi
        ON o.order_id = oi.order_id
GROUP BY u.user_id, u.name, u.channel
ORDER BY 총구매액 DESC;
```

> 💡 **조인 순서대로 읽기**
> users → (그 회원의) orders → (그 주문의) order_items : 키를 따라 한 단계씩 연결함.
> 4개 테이블이면 ON 조건 3개가 필요하다(N개 테이블 → N-1개 조인 조건).

## 6. JOIN 함정 (반드시 이해)

### 6-1. 행 증식 (Fan-out)
한 주문에 상품이 3개면, `orders ⋈ order_items`는 그 주문이 **3행으로 늘어난다**. 이 상태에서 `orders` 쪽 값을 `SUM`하면 **중복 합산**된다.

```sql
-- ⚠️ 위험: 주문 금액을 order_items와 조인 후 잘못 집계하면 부풀려짐
-- 안전: 회원별 주문 '건수'는 COUNT(DISTINCT o.order_id)로 중복 제거
SELECT
    u.name,
    COUNT(DISTINCT o.order_id) AS 주문수,        -- DISTINCT 필수
    SUM(oi.quantity * oi.unit_price) AS 총구매액 -- 라인 단위 합은 OK
FROM users u
JOIN orders o ON u.user_id = o.user_id
JOIN order_itmes oi ON o.order_id = oi.order_id
GROUP BY u.user_id, u.name;
```

> ⚠️ **핵심 규칙**
> LEFT JOIN에서 **오른쪽 테이블 조건은 `ON`에**, 왼쪽(기준) 테이블 조건은 `WHERE`에. `WHERE o.컬럼`을 쓰면 NULL이 탈락해 사실상 INNER JOIN이 된다. 초보자가 가장 많이 틀리는 부분.

### 6-2. LEFT JOIN에서 WHERE의 함정
```sql
-- ❌ 의도와 다름: LEFT JOIN인데 WHERE로 오른쪽을 걸면 INNER JOIN이 되어버림
SELECT u.name, o.order_id
FROM users u
LEFT JOIN orders o ON u.user_id = o.user_id
WHERE o.status = 'completed';   -- 주문 없는 회원(NULL)이 여기서 탈락!

-- ✅ 의도대로: 필터 조건을 ON으로 옮기면 LEFT 보존
SELECT u.name, o.order_id
FROM users u
LEFT JOIN orders o
       ON u.user_id = o.user_id
      AND o.status = 'completed';   -- 회원은 모두 남고, 완료 주문만 붙음
```

> ⚠️ **핵심 규칙**
> LEFT JOIN에서 **오른쪽 테이블 조건은 `ON`에**, 왼쪽(기준) 테이블 조건은 `WHERE`에. `WHERE o.컬럼`을 쓰면 NULL이 탈락해 사실상 INNER JOIN이 된다. 초보자가 가장 많이 틀리는 부분.

## 7. SELF JOIN & CORSS JOIN

### 7-1. SELF JOIN - 같은 테이블을 자기 자신과 조인

```sql
-- 같은 채널로 가입한 회원 쌍 찾기 (자기 자신끼리 비교)
SELECT
    a.name AS 회원A,
    b.name AS 회원B,
    a.channel
FROM users a
JOIN users b
   ON a.channel = b.channel
  AND a.user_id < b.user_id     -- 중복쌍·자기자신 제거
ORDER BY a.channel;
```

> 조직도(상사-부하), 추천인 관계 등 "같은 테이블 안의 행끼리 관계"를 볼 때 사용.

### 7-2. CROSS JOIN — 모든 조합 (카티션 곱)

```sql
-- 모든 회원 × 모든 카테고리 조합 (예: 분석용 빈 격자 만들기)
SELECT u.name, c.category
FROM users u
CROSS JOIN (SELECT DISTINCT category FROM products) c;
```

> ⚠️ 행 수 = 왼쪽 × 오른쪽
> 의도치 않은 CROSS JOIN(=ON 조건 누락)은 폭발적 행 증가의 원인.
> "조회 결과가 비정상적으로 많다" 싶으면 조인 조건부터 확인.

## 8. USING/NATURAL JOIN (참고)

```sql
-- 양쪽 조인 컬럼명이 같으면 USING으로 간결하게
SELECT u.name, o.order_id
FROM users u
JOIN orders o USING (user_id);     -- ON u.user_id = o.user_id 와 동일

-- NATURAL JOIN: 같은 이름 컬럼을 자동 조인 (편하지만 위험 → 실무 비권장)
```

> 💡 `USING`은 깔끔하지만, `NATURAL JOIN`은 "같은 이름 컬럼"을 멋대로 다 조인해서 의도치 않은 결과를 낳을 수 있다. **명시적 `ON`을 기본**으로 쓰는 걸 권장.

## 9. 종합 예시 (실무형)

```sql
-- "채널별 매출 & 객단가(AOV) 리포트 - 완료 주문 기준"
SELECT
    u.channel AS 채널,
    COUNT(DISTINCT u.user_id)           AS 구매회원수,
    COUNT(DISTINCT o.order_id)          AS 주문수,
    SUM(oi.quantity * oi.unit_price)    AS 총매출,
    ROUND(
        SUM(oi.quantity * oi.unit_price)::NUMERIC
        / NULLIF(COUNT(DISTINCT o.order_id), 0)
    ) AS 객단가_AOV
FROM users u
JOIN orders o
  ON u.user_id = o.user_id
 AND o.status = 'completed'
JOIN order_itmes oi
   ON o.order_id = oi.order_id
GROUP BY u.channel
ORDER BY 총매출 DESC;
```

### 🟡 Lv.2 — 응용
 
**Q4.** 회원별 주문 건수를 구하되, **주문이 없는 회원은 0으로** 표시하시오. (`LEFT JOIN` + `COUNT(컬럼)`)
```sql
SELECT u.name, COUNT(o.order_id) AS 주문건수

LEFT JOIN orders o ON u.user_id = o.user_id
GROUP BY u.user_id, u.name
ORDER BY 주문건수 DESC;
```


**Q5.** **한 번도 주문하지 않은 회원**의 이름과 채널을 조회하시오.
```sql
SELECT u.name, u.channel
FROM users u
LEFT JOIN orders o ON u.user_id = o.user_id
WHERE order_id IS NULL;
```


**Q6.** 카테고리별 총 매출(`quantity * unit_price` 합)과 판매된 상품 종류 수를, 매출 높은 순으로 조회하시오. (`order_items` ⋈ `products`)
```sql
SELECT
    p.category,
    SUM(oi.quantity * oi.unit_price) AS 총매출,
    COUNT(DISTINCT p.product_id) AS 상품종류수
FROM order_items oi 
LEFT JOIN products p ON p.product_id = oi.product_id
GROUP BY p.category
ORDER BY 총매출 DESC;
```

**Q7.** 회원별 총 구매액을 구하되, **완료(`completed`) 주문만** 합산하시오. 단, 완료 주문이 없는 회원도 결과에 포함하고 구매액 0으로 표시하시오. (LEFT JOIN + ON 조건 위치 주의)
```sql
SELECT
	u.name,
	COALESCE(sum(oi.quantity * oi.unit_price), 0) AS 총구매액
FROM users u
LEFT JOIN orders o
	ON u.user_id = o.order_id
   AND o.status = 'completed'
LEFT JOIN order_items oi
	ON o.order_id = oi.order_id
GROUP BY u.user_id, u.name
ORDER BY 총구매액 DESC;
```

### 🔴 Lv.3 — 포트폴리오급
 
**Q8. [분석 미션] "채널별 매출 성과 & 전환 리포트"**
 
경영팀 요청:
 
> "획득 채널별로 ① 전체 회원 수 ② 구매 경험 회원 수(완료 주문 보유) ③ 구매 전환율(%) ④ 총매출 ⑤ 객단가(AOV)를 한 테이블로요. 매출 높은 순, 컬럼명 한글로."
 
**요구사항**
1. `users` 기준 LEFT JOIN으로 전체 회원 보존
2. 완료 주문 필터는 `ON`에 배치 (위 함정 적용)
3. 구매 전환율 = 구매회원수 / 전체회원수, 객단가 = 매출 / 주문수 (둘 다 `::NUMERIC` + `NULLIF`)
4. fan-out 주의: 주문수는 `COUNT(DISTINCT o.order_id)`, 구매회원수는 `COUNT(DISTINCT o.user_id)`
5. `phase4_channel_performance.sql` 저장 + 비즈니스 질문 주석 + GitHub 커밋

```sql

```
