# Phase 1-4: 조인 (JOIN)

> **목표**: 여러 테이블을 **연결**해 진짜 비즈니스 지표를 뽑는다. "회원별 총 구매액", "카테고리별 매출", "한 번도 주문 안 한 회원" 같은 분석이 가능해진다. 실무·SQLD 최빈출 구간.
> **선수 조건**: Phase 1~3 완료 (`GROUP BY`, 조건부 집계 자유 구사)
> **SQLD 연계**: 2과목 — 조인의 종류, 표준 조인, 카티션 곱


## 0. 이 페이지에서 얻어갈 것

- [ ] `INNER JOIN`과 `LEFT JOIN`의 차이를 그림으로 설명
- [ ] `ON`(조인 조건)과 `WHERE`(필터)의 역할 구분
- [ ] JOIN + `GROUP BY`로 회원별/카테고리별 지표 산출
- [ ] `LEFT JOIN ... IS NULL`로 "없는 것" 찾기
- [ ] SELF JOIN / CROSS JOIN의 쓰임
- [ ] **JOIN 함정**: 행 증식(fan-out), LEFT JOIN에서 WHERE의 함정


## 1. JOIN이 왜 필요한가

Phase 0에서 데이터를 정규화해 여러 테이블로 나눔. (회원·주문·상품)
"회원 이름 + 그 회원의 주문 매출"을 함께 보려면 테이블을 다시 연결해야 함.
연결 도구가 [JOIN]

```
users          orders              order_items         products
user_id 🔑 ──< user_id 🔗         order_id 🔗   >──  order_id 🔑
               order_id 🔑 ──< ... product_id 🔗 ──>  product_id 🔑
```

연결고리(키)로 테이블을 이음.
`users.users_id = orders.user_id`, `orders.order_id = order_items.order_id`,
`order_items.product_id = products.product_id`


## 2. INNER JOIN - 양쪽 모두 있는 것만

```sql
-- 주문한 회원의 이름과 주문 정보 (양쪽에 매칭되는 행만)
SELECT
    u.name       AS 회원명,
    o.order_id   AS 주문번호,
    o.order_date AS 주문일,
    o.status     AS 상태
FROM users AS u
INNER JOIN orders AS o
        ON u.user_id = o.user_id        -- 연결 조건
ORDER BY o.order_date;
```

> 💡 **별칭(alias) 습관**
> 테이블에 `u`, `o` 같은 별칭을 붙이고 컬럼 앞에 `u.`, `o.`를 명시함.
> 여러 테이블에 같은 컬럼명(`user_id`)이 있을 때 모호함을 없애고 가독성도 좋음.
 
`INNER JOIN`은 **양쪽 테이블에서 조인 조건이 맞는 행만** 남긴다.
주문이 없는 회원은 결과에서 빠짐.


## 3. LEFT JOIN - 왼쪽은 모두 보존

```sql
-- 모든 회원 + (있으면) 주문. 주문 없는 회원도 NULL로 표시
SELECT
    u.name,
    o.order_id,
    o.status
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
ORDER BY u.user_id;
-- 주문 없는 회원(예: 임채원)은 order_id가 NULL로 나옴.
```

| 조인 | 결과 |
|------|------|
| `INNER JOIN` | 양쪽 매칭되는 행만 (교집합) |
| `LEFT JOIN` | 왼쪽 전부 + 매칭되는 오른쪽 (없으면 NULL) |
| `RIGHT JOIN` | 오른쪽 전부 + 매칭되는 왼쪽 |
| `FULL OUTER JOIN` | 양쪽 전부 (매칭 안 되면 각각 NULL) |

> 💡 **실무 팁**
> `LEFT JOIN`이 가장 많이 쓰임.
> "기준 테이블(회원/상품)은 다 보여주고, 연관 데이터를 붙인다"는 분석 패턴에 맞기 때문.
> `RIGHT JOIN`은 `LEFT`로 바꿔 쓸 수 있어 거의 사용하지 않음.

### "없는 것" 찾기 - LEFT JOIN + IS NULL

```sql
-- 한 번도 주문하지 않은 회원 (이탈/미활성 회원 발굴)
SELECT u.user_id, u.name, u.channel
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
WHERE o.order_id IS NULL;           -- 매칭 안 된 행 = 주문 없음
```

> 💡 매우 자주 쓰는 패턴
> "구매 전환 안 한 회원", "리뷰 없는 상품" 등 **부재(absence)** 를 찾을 때.


## 4. JOIN + GROUP BY - 진짜 지표의 시작 ⭐
이게 Phase 3과 Phase 4가 만나는 지점이자, 실무 분석의 핵심.

```sql
-- 회원별 주문 건수 (주문 없는 회원도 0으로 표시)
SELECT
    u.name,
    COUNT(o.orderid) AS 주문건수    -- COUNT(컬럼)은 NULL 제외 → 주문 없으면 0
FROM users AS u
LEFT JOIN orders AS o
       ON u.user_id = o.user_id
GROUP BY u.user_id, u.name          -- PK까지 묶어야 동명이인 안전
ORDER BY 주문건수 DESC;
```

> ⚠️ `COUNT(o.order_id)`와 `COUNT(*)`의 차이
> LEFT JOIN에서 주문 없는 회원은 `o.order_id`가 NULL이라 `COUNT(o.order_id)=0`
> 하지만 `COUNT(*)`는 그 NULL 행도 1로 세어 **1**이 된다. 주문 건수를 정확히 세려면 `COUNT(컬럼)`.

```sql
-- 카테고리별 총 매출 (3개 테이블 조인)
SELECT
    p.category,
    SUM(oi.quantity * oi.unit_price) AS 총매출,
    COUNT(DISTINCT oi.order_id)      AS 주문수
FROM order_items AS oi
INNER JOIN products AS p
    ON oi.product_id = p.product_id
GROUP BY p.category
ORDER BY 총매출 DESC;
```


## 5. 다중 테이블 조인 (4개 연결)

```sql
-- "회원별 총 구매액" - 유효 주문(completed)만, 매출 높은 순
SELECT
    u.name AS 회원명,
    u.channel AS 채널,
    COUNT(DISTINCT o.order_id)       AS 주문수,
    SUM(oi.quantity * oi.unit_price) AS 총구매액
FROM users AS u
INNER JOIN orders AS o
        ON u.user_id = o.user_id
       AND o.status = 'completed'       -- 조인 시점에 상태 필터 (4장 함정 참고)
INNER JOIN order_items AS oi
        ON o.order_id = oi.order_id
GROUP BY u.user_id, u.name, u.channel
ORDER BY 총구매액 DESC;
```

> 💡 **조인 순서대로 읽기**
> users → (그 회원의) orders → (그 주문의) order_items : 키를 따라 한 단계씩 연결함.
> 4개 테이블이면 ON 조건 3개가 필요하다(N개 테이블 → N-1개 조인 조건).
